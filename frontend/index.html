    Reference Guide — Field Mapping & Flow

    📋 How this tool works

    This tool generates a VBKREQ XML (Vendor Booking Request) for submission to E2open TA and DavisTurner. The UI is split into two independent sections:

    📦 Booking — Steps 1 & 2

    📊 1 · Upload Supplier Excel
    →
    📡 2 · Fetch Databricks ASN
    →
    🗂 3 · Build Master Workbook
    →
    ⚡ 4 · Generate VBKREQ XML
    →
    🚀 5 · Upload to E2open SFTP

    ✕ Cancellation — standalone, no re-upload needed

    🔍 Enter PO or VB Ref
    →
    📂 Look up stored booking data
    →
    ⚡ Build cancellation VBKREQ (01 + 177)
    →
    🚀 Upload to E2open SFTP

    Step 1 — Supplier Excel (two-sheet template): The supplier fills in the downloadable template. The template has two data sheets: PO Header (one row per PO — mandatory/defaulted booking fields) and PO Lines (one row per SKU — quantities, weights, carton details). A FACTORY_LOOKUP tab lists valid factory codes; selecting a Factory_ID auto-fills name and address via VLOOKUP. Mandatory (pink) columns must be completed; defaulted (green) columns are pre-set and can be changed if needed. The PO Header sheet pre-fills Traffic_Mode = CFS and Booking_Group = Single Booking as defaults. Mode_Of_Transport is sourced from Databricks and is optional in the template.

    Booking_Group controls how POs are consolidated into bookings: Single Booking = one VBKREQ per PO; Multiple POs-BK001…BK025 = all POs sharing the same code are merged into one VBKREQ; Multiple = all POs in the upload combined into a single VBKREQ.

    Factory_ID 9999 (Dummy Factory): If a real factory code is not yet known, use 9999. Address validation is skipped for this code — the VBKREQ will be generated without a factory address.

    Step 2 — Booking Pipeline (single button): Step 2 is always visible but locked/dimmed until Step 1 completes successfully. Select Purpose Code (13 = New Submission, 15 = Re-Submission) then click ⚡ Build VB & Upload to SFTP. All four stages (Fetch ASN → Build → Generate → Upload) run automatically and results appear in a single table: PO | ASN Ref(s) | VB Ref | VBKREQ File download. A tagged supplier template with prefilled booking refs is also available for download per result row.

    Recent Booking History: The collapsible 🕑 panel below Step 2 shows all VBKREQs generated in the last 3 days, loaded automatically after each pipeline run. Click to expand.

    Cancel Booking (standalone): Enter one or more PO numbers or VB Refs (e.g. 500034236929 or VB-1000000025) in the Cancel Booking card at the bottom. Click Look Up Existing Bookings to verify the records exist in the generation log and confirm ✅ Ready to cancel status. Then click Cancel & Upload to SFTP — the system builds a VBKREQ with PurposeCd=01, reuses the original VB Ref, injects DateTypeCd=177 (cancellation status date), and uploads. No supplier template re-upload is required.

    ⚠ Legacy bookings: Bookings created before the current version of the tool will show ⚠ No stored data (re-run booking first) in the lookup panel. To enable cancellation for these, re-run the pipeline with the original supplier template once to refresh the stored master row data in the generation log.

    ✕ Cancel Booking — Standalone Cancellation Flow
    StepDetail
    1 · Enter inputType one or more PO numbers (e.g. 500034236929) or VB Refs (e.g. VB-1000000025) — comma or newline separated. Both types can be mixed in one lookup.
    2 · Look UpFetches matching entries from the generation log (bible/generation-log.json). Displays: Input | PO Number(s) | VB Ref | Booked On. A ✅ Ready to cancel badge confirms stored master row data is present; ⚠ No stored data means the booking predates the current version — re-run the pipeline first to refresh.
    3 · Cancel & UploadBuilds a VBKREQ XML with PurposeCd=01, reuses the original Booking_Ref (VB-XXXXX) from the log, injects DateTypeCd=177 (cancellation status date) immediately after the SBK date, and uploads to E2open SFTP. No supplier template re-upload required.
    Generation LogEvery successful pipeline run saves the full master row data alongside the VB Ref and timestamp in bible/generation-log.json. This makes cancellation fully self-contained — the Cancel Booking card is independent of the current session state.
